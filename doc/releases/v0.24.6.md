# Release Notes v0.24.6
<!-- markdownlint-disable MD013 MD060 -->

**Release Date:** 2026-05-03
**Type:** Patch Release (Bug Fix)

## Overview

Version 0.24.6 fixes two bugs introduced or exposed by the 0.24.5 release:

1. The installer corrupts itself when updating from the install prefix - bash reads
   scripts lazily and `cp` was overwriting the running script at the same inode.
2. `oradba_standard.conf` uses `${ORACLE_SID,,}` (bash 4.0+) in an evaluated context,
   causing `bad substitution` when macOS system bash 3.2 sources the startup chain.

---

## Fixed

### Installer: Self-Overwrite Corruption During Update

**Symptom:** Running `oradba_install.sh --github` from the installed location
(`/opt/oracle/local/oradba/bin/oradba_install.sh`) fails intermittently with:

```text
[INFO] Installing files...
/opt/oracle/local/oradba/bin/oradba_install.sh: line 2207: syntax error near unexpected token `('
```

**Root cause:** Bash reads scripts lazily from an open file descriptor. When the
installer executes `cp -r "$TEMP_DIR"/* "$INSTALL_PREFIX/"`, it overwrites its own
file at the same inode. Bash then continues reading from the old byte offset into the
new file content, hitting unexpected tokens. The second attempt "works" because the
new version was partially installed on the first run.

**Fix:** Before the file copy phase, the running script is moved to a temporary name
via `mv`. `mv` changes only the directory entry (rename syscall) without touching the
inode, so bash's open fd stays valid. The `cp` then creates a new inode for the
updated installer. The old temp file is removed after installation completes.

```bash
# Before copy: protect running script from inode-level overwrite
if [[ "${BASH_SOURCE[0]:-}" == "${INSTALL_PREFIX}"/* ]]; then
    _SELF_GUARD="${BASH_SOURCE[0]}.updating"
    mv "${BASH_SOURCE[0]}" "${_SELF_GUARD}"
fi
cp -r "$TEMP_DIR"/* "$INSTALL_PREFIX/"
[[ -n "${_SELF_GUARD}" ]] && rm -f "${_SELF_GUARD}"
```

Additionally, the hidden-file copy (`.[!.]*`) was rewritten as an explicit `for` loop
with a `[[ -e ]]` guard, eliminating a potential unmatched-glob issue when no hidden
files exist in the temporary directory.

### `oradba_standard.conf`: `${ORACLE_SID,,}` Bad Substitution on macOS

**Symptom:** VS Code integrated terminal shows the following on shell startup:

```text
INFO: sourcing OraDBA environment...
bash: ${ORACLE_BASE}/diag/rdbms/${ORACLE_SID,,}/${ORACLE_SID}: bad substitution
```

**Root cause:** Line 404 of `oradba_standard.conf` used `${ORACLE_SID,,}` (bash 4.0+
lowercase parameter expansion) inside a double-quoted `export` assignment that is
evaluated at source time. When macOS system bash (`/bin/bash`, version 3.2) is the
initial login shell - which happens when `terminal.integrated.defaultProfile.osx` is
not explicitly set in VS Code settings - the substitution fails.

The misleading part: running `bash --version` in the already-open terminal shows
homebrew bash 5.x because that command starts a new subshell from PATH. The shell
that sourced `~/.bash_profile` was already system bash 3.2.

**Fix:** Replaced `${ORACLE_SID,,}` with a `tr`-based equivalent:

```bash
# Before (bash 4.0+ only):
export ORADBA_ORA_DIAG_SID="${ORACLE_BASE}/diag/rdbms/${ORACLE_SID,,}/${ORACLE_SID}"

# After (bash 3.2 compatible):
export ORADBA_ORA_DIAG_SID="${ORACLE_BASE}/diag/rdbms/$(printf '%s' "${ORACLE_SID}" | tr 'A-Z' 'a-z')/${ORACLE_SID}"
```

**VS Code workaround:** Set an explicit default terminal profile to avoid system bash:

```json
"terminal.integrated.defaultProfile.osx": "Enterprise (bash)"
```
