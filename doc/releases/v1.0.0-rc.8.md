# Release Notes v1.0.0-rc.8

<!-- markdownlint-disable MD013 MD060 -->

**Release Date:** 2026-07-07
**Type:** Release Candidate (Bugfix RC - M10)
**Soak window:** 2026-07-07 to 2026-07-10 (same target as rc.1 through rc.7)

## Overview

Version 1.0.0-rc.8 fixes a regression in the in-place update path (`--update --github`):
the self-guard `mv` failed because the installer file had already been deleted during the
update's old-installation removal step. Also adds `ORADBA_PS1_DIRTRIM` support for
trimming deep `\w` paths in the shell prompt. No API changes.

---

## What Changed Since rc.7

### Fixed - symlinks in `etc/` deleted during `--update`

- **`src/bin/oradba_install.sh`** `preserve_runtime_files`: user symlinks in
  `etc/` (e.g. `oradba_customer.conf -> ~/.../dotfiles/oradba_customer.conf`,
  `oradba_homes.conf -> ~/.../dotfiles/oradba_homes.conf`,
  `datasafe.conf -> ~/.../dotfiles/projects/.../datasafe.conf`) were silently
  dropped during `--update --github`. Root cause: the `find` for runtime files
  used `-type f`; symlinks (`-type l`) were invisible. All three are deleted by
  `rm -rf "$install_dir"` in the update flow and were never re-created.

  Fix: extended both `find` calls (core `etc/` and `extensions/*/etc/`) to also
  match `-type l`. All symlinks in `etc/` are user-created (the payload never
  ships symlinks). Also added `*_customer.conf` to the regular-file preserve
  pattern for non-symlinked user configs.

- **`src/bin/oradba_install.sh`** `restore_runtime_files`: `find -type f` missed
  symlinks saved in the temp preserve directory; changed to
  `\( -type f -o -type l \)`. `cp -P` was already in place, so the symlink is
  recreated correctly (not followed).

- **`src/bin/oradba_extension.sh`** `update_extension`: user-added symlinks in
  extension `etc/` not detected or saved by the user-file scan (`find -type f`).
  Changed to `\( -type f -o -type l \)`, save uses `cp -P`, restore uses `cp -P`.

### Fixed - `mv` failure during `--update --github`

- **`src/bin/oradba_install.sh`** self-guard block: added `[[ -f "${BASH_SOURCE[0]}" ]]`
  guard before `mv "${BASH_SOURCE[0]}" "${BASH_SOURCE[0]}.updating"`.

  **Root cause:** `--update --github` first removes the old installation with
  `rm -rf "$install_dir"` (which deletes `bin/oradba_install.sh`), then later creates
  a fresh empty `$INSTALL_PREFIX`. When the self-guard code runs, `BASH_SOURCE[0]`
  no longer exists on disk and `mv` fails:

  ```text
  mv: rename …/bin/oradba_install.sh to …/bin/oradba_install.sh.updating: No such file or directory
  ```

  The guard is only needed when `cp` would overwrite the running script at the same
  inode. If the file is already gone (update path), the guard is unnecessary and is
  now skipped.

### Added - `ORADBA_PS1_DIRTRIM` prompt path trimming

- **`src/etc/oradba_core.conf`**: new `ORADBA_PS1_DIRTRIM` variable (default: `0`,
  off). When set to a positive integer, the prompt `\w` is trimmed to the last N
  path components via bash `PROMPT_DIRTRIM`. Requires bash 4.0+; silently ignored on
  older bash. Useful for deep paths (e.g. OneDrive customer folders).
- **`src/etc/oradba_standard.conf`**: apply `PROMPT_DIRTRIM` when
  `ORADBA_PS1_DIRTRIM > 0`.

## How to Promote to v1.0.0

After the soak window (not before 2026-07-10):

1. Confirm no Critical/High regressions filed against v1.0.0-rc.8
2. Run Docker integration test and record results in `doc/v1.0.0-readiness.md`
3. Mark remaining open items `[x]` in `doc/v1.0.0-readiness.md`
4. Set `VERSION` to `1.0.0` and add `## [1.0.0]` date to `CHANGELOG.md`
5. Create `doc/releases/v1.0.0.md`
6. Check `doc/review/risk-register.md` and `technical-debt-register.md` — open
   items become GitHub issues, then delete `doc/review/`.
7. Tag: `git tag v1.0.0 HEAD`
8. Push tag and let the release workflow create the GitHub Release
