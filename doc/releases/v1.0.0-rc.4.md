# Release Notes v1.0.0-rc.4

<!-- markdownlint-disable MD013 MD060 -->

**Release Date:** 2026-06-29
**Type:** Release Candidate (Bugfix RC - M10)
**Soak window:** 2026-06-29 to 2026-07-10 (same target as rc.1/rc.2/rc.3)

## Overview

Version 1.0.0-rc.4 is a pure bugfix release on top of rc.3. It fixes the root
cause of the Data Safe connector status reporting regression: Oracle CMAN
processes rename `argv[0]` at startup, making `ps -ef | grep "${path}.*cmadmin"`
permanently unreliable. Status detection is now port-based and definitive.
Also fixes a secondary LD_LIBRARY_PATH pollution issue when checking multiple
connectors in the same session. No new features, no API changes.

---

## What Changed Since rc.3

### Fixed - Data Safe connector status (root cause)

- **`plugin_check_status()` (`src/lib/plugins/datasafe_plugin.sh`)**: replaced
  broken process-based (ps-grep) detection with port-based status check.

  **Root cause:** Oracle CMAN processes (`cmadmin` / `cmgw`) rename `argv[0]`
  to just the binary name without the installation path using `prctl`/argv
  manipulation. Therefore `ps -ef | grep "${path}.*cmadmin"` never matches a
  running CMAN process - the ps-based fallback was permanently dead code on
  production systems.

  **New detection order:**

  1. **cmctl show services** - positive indicators (`Services Summary`, `READY`,
     `started`, `is running`) return 0 immediately. TNS errors are ignored
     (IPC unreachable does not mean stopped). Non-TNS stopped indicators deferred.
  2. **Port-based check** (`_datasafe_port_listening()`) - reads the configured
     port from `cman.ora` via `plugin_get_port()`, then checks whether that port
     is actively listening using `ss` (Linux), `lsof` (macOS/Linux), or `netstat`
     as fallback. A listening port is authoritative proof the connector is running;
     a closed port confirms it is stopped. Immune to `argv[0]` renaming, systemd
     IPC isolation, and stale process caches.
  3. **Python `setup.py`** - only reached when no port is configured in `cman.ora`.

  Added `TRACE`-level logging at every decision point for full observability
  (`ORADBA_LOG_LEVEL=trace oradba_dsctl.sh status <name>`).

  `_datasafe_port_listening()` is defined as a named function so the test suite
  can override it with a one-line stub; no real listeners required in unit tests.

- **`ORADBA_CACHED_PS` stale-snapshot cleanup (two components from rc.3 work):**
  - `show_status()` in `oradba_dsctl.sh`: unsets `ORADBA_CACHED_PS` before
    calling the plugin so `oradba_dsctl.sh status` always performs a fresh check.
  - `show_oracle_status_registry()` in `oraup.sh`: unsets `ORADBA_CACHED_PS` at
    function exit so the batch snapshot does not persist in the parent shell when
    `oraup.sh` is sourced via the `u` alias; previously the stale export caused
    subsequent `oradba_dsctl.sh status` calls to report RUNNING for stopped
    connectors.

### Fixed - LD_LIBRARY_PATH per-connector isolation

- **`execute_plugin_function_v2()` (`src/lib/oradba_common.sh`)**: always sets
  `LD_LIBRARY_PATH="${oracle_home}/lib"` for each plugin subshell. Previously
  guarded by `if [[ -z "${LD_LIBRARY_PATH:-}" ]]`, which caused all connectors
  after the first to use the first connector's library path when the calling
  shell had `LD_LIBRARY_PATH` already set (e.g. from a profile that sources one
  connector's environment). Each plugin subshell now always starts with the
  correct connector-specific library path.

## How to Promote to v1.0.0

After the soak window (not before 2026-07-10):

1. Confirm no Critical/High regressions filed against v1.0.0-rc.4
2. Run Docker integration test and record results in `doc/v1.0.0-readiness.md`
3. Mark remaining open items `[x]` in `doc/v1.0.0-readiness.md`
4. Set `VERSION` to `1.0.0` and add `## [1.0.0]` date to `CHANGELOG.md`
5. Create `doc/releases/v1.0.0.md`
6. Check `doc/review/risk-register.md` and `technical-debt-register.md` - open
   items become GitHub issues, then delete `doc/review/`.
7. Tag: `git tag v1.0.0 HEAD`
8. Push tag and let the release workflow create the GitHub Release
