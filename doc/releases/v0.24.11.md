# Release Notes v0.24.11
<!-- markdownlint-disable MD013 MD060 -->

**Release Date:** 2026-06-25
**Type:** Patch Release (Bug Fix)

## Overview

Version 0.24.11 fixes a crash in `oradba_dsctl.sh` when the log directory
(`/var/log/oracle` by default) does not exist. Under `set -euo pipefail` any log
write via `oradba_common.sh` immediately aborts the script with "No such file or
directory", causing the systemd service unit to fail at `ExecStart`.

---

## Fixed

### `oradba_dsctl.sh`: crash when log directory is missing

**Symptom.** The Data Safe connector service fails to start and `journalctl` shows:

```text
/opt/oradba/bin/oradba_dsctl.sh: line NNN: /var/log/oracle/oradba_dsctl.log:
No such file or directory
```

**Root cause.** `oradba_dsctl.sh` sets its log file as:

```bash
LOGFILE="${ORADBA_LOG:-/var/log/oracle}/${SCRIPT_NAME%.sh}.log"
```

`oradba_common.sh` writes every log line with `echo >> "${ORADBA_LOG_FILE}"`.
When `/var/log/oracle/` does not exist (e.g. fresh system, non-root service user,
or `ORADBA_LOG` pointing to a path that was never created), the first log write
fails under `set -euo pipefail` and the script exits.

**Fix.** After computing `LOGFILE`, the script ensures the directory exists:

```bash
if [[ ! -d "${LOGFILE%/*}" ]]; then
    mkdir -p "${LOGFILE%/*}" 2>/dev/null || LOGFILE="/tmp/${SCRIPT_NAME%.sh}.log"
fi
```

If `mkdir -p` succeeds, logging works as before. If it fails (insufficient
permissions), `LOGFILE` falls back to `/tmp/oradba_dsctl.log` — logging continues
and execution is not interrupted.

**Companion fix.** `odb_datasafe` v0.20.4 sets
`Environment="ORADBA_LOG=${CONNECTOR_HOME}/log"` in generated service files.
This points the log into the connector's own home tree (which always exists after
install) and avoids the system directory dependency entirely for managed connectors.

## Verification performed

- `shellcheck --shell=bash src/bin/oradba_dsctl.sh` clean
- `make lint` passes
- Script starts correctly with and without `/var/log/oracle/` present
- With permission denied on log dir: falls back to `/tmp/oradba_dsctl.log`, no crash

---

## Upgrade notes

No configuration changes required. The fix is transparent: existing environments
where `/var/log/oracle/` is present continue to log there. Environments where the
directory is absent now use `/tmp` as a fallback instead of crashing.

For managed Data Safe connectors, re-running `--prepare` and `--install` in
`odb_datasafe` v0.20.4 will add `ORADBA_LOG=${CONNECTOR_HOME}/log` to the service
file and create the log directory under the connector home.
