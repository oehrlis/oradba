# Release Notes v1.0.0-rc.3

<!-- markdownlint-disable MD013 MD060 -->

**Release Date:** 2026-06-29
**Type:** Release Candidate (Bugfix RC - M10)
**Soak window:** 2026-06-29 to 2026-07-10 (same target as rc.1/rc.2)

## Overview

Version 1.0.0-rc.3 is a pure bugfix release on top of rc.2. It fixes four
issues discovered during production validation of BasEnv coexistence mode and
the Data Safe on-premises connector with systemd. No new features, no API
changes. All 1713 tests pass.

---

## What Changed Since rc.2

### Fixed - Login sourcing

- **`execute_plugin_function_v2()` (`src/lib/oradba_common.sh`)**: `mktemp`
  failed during early macOS login sourcing because the per-session `$TMPDIR`
  directory (`/var/folders/...`) does not exist yet when `launchd` sources the
  shell profile. The function logged `[ERROR]` and aborted the plugin call.
  Now tries `TMPDIR=/tmp mktemp` as a fallback; if both fail, redirects plugin
  stderr to `/dev/null` with a DEBUG message and continues normally.

### Fixed - BasEnv coexistence mode

- **`update_profile()` (`src/bin/oradba_install.sh`)**: the manual-integration
  block ("Add the following block to ... AFTER the BasEnv init line") was only
  printed when `--update-profile` was explicitly passed. It now always appears
  in BasEnv coexistence mode. The `[WARN]` about `--update-profile` not being
  supported remains conditional on the flag actually being specified.

- **`$etc` / `$log` short variables (`src/etc/oradba_standard.conf`)**: both
  variables are also exported by BasEnv (from `ETC_BASE` / its log directory).
  In coexistence mode oradba unconditionally overwrote them with oradba's own
  paths, breaking BasEnv aliases such as `sqh`, `rmanh`, and `rmanch` that pass
  `$etc/sqlplus.filter` and `$etc/sqlplus.key` to `rlwrap`. Both exports are
  now guarded by `ORADBA_COEXIST_MODE` and only run in standalone mode.

### Fixed - Data Safe / systemd connector status

- **`plugin_check_status()` (`src/lib/plugins/datasafe_plugin.sh`)**: when a
  Data Safe on-premises connector is started by systemd oneshot, `cmctl show
  services` cannot reach the CMAN admin IPC socket from an unrelated session
  and returns a `TNS-*` error. The previous code treated any `TNS-` or
  `stopped` match as an early return 1 before process detection ran.

  New detection order:
  1. Positive cmctl indicators (`Services Summary`, `READY`, `started`,
     `is running`) → return 0 immediately.
  2. TNS errors → ignored (unreachable ≠ stopped); fall through to process
     detection.
  3. Non-TNS stopped indicators (`not running`, `stopped`) → deferred; act as
     final verdict only when process detection also finds nothing.
  4. Process detection (`cmadmin` / `cmgw` in process list) → return 0 if found.

  Side effect fixed: `oradba_dsctl.sh start` no longer calls `cmctl startup`
  on an already-running connector (which produced a misleading "started
  successfully" on the first call and an error on the second).

## How to Promote to v1.0.0

After the soak window (not before 2026-07-10):

1. Confirm no Critical/High regressions filed against v1.0.0-rc.3
2. Run Docker integration test and record results in `doc/v1.0.0-readiness.md`
3. Mark remaining open items `[x]` in `doc/v1.0.0-readiness.md`
4. Set `VERSION` to `1.0.0` and add `## [1.0.0]` date to `CHANGELOG.md`
5. Create `doc/releases/v1.0.0.md`
6. Check `doc/review/risk-register.md` and `technical-debt-register.md` — open
   items become GitHub issues, then delete `doc/review/`.
7. Tag: `git tag v1.0.0 HEAD`
8. Push tag and let the release workflow create the GitHub Release
