# Release Notes v1.0.0-rc.5

<!-- markdownlint-disable MD013 MD060 -->

**Release Date:** 2026-06-30
**Type:** Release Candidate (Bugfix + Feature RC - M10)
**Soak window:** 2026-06-30 to 2026-07-10 (same target as rc.1 through rc.4)

## Overview

Version 1.0.0-rc.5 builds on rc.4 and fixes two remaining Data Safe connector
management issues: a silent script abort caused by `set -e` when checking a
stopped connector, and a systemd state mismatch after manual `oradba_dsctl.sh`
stop/start calls. It also adds a `services` subcommand and `dssvc` alias to list
installed connector service units. No API changes, no schema changes.

---

## What Changed Since rc.4

### Fixed - `oradba_dsctl.sh status` silent exit

- **`show_status()` / `start_connector()` / `stop_connector()` (`src/bin/oradba_dsctl.sh`)**:
  fix silent script abort under `set -euo pipefail` when `execute_plugin_function_v2`
  returns exit code 1 (connector stopped).

  **Root cause:** The call and the exit-code capture were on separate lines:

  ```bash
  # Before (broken): set -e exits the script here when connector is stopped
  execute_plugin_function_v2 ... > /dev/null 2>&1
  status_exit_code=$?
  ```

  With `set -euo pipefail` active, a non-zero return from
  `execute_plugin_function_v2` caused the script to exit immediately. The
  assignment `status_exit_code=$?` was never reached, `show_status()` produced
  no output, and the script exited with code 1 - giving the appearance of a
  crash rather than a stopped connector.

  **Fix:** All three call sites now use the `||` idiom:

  ```bash
  execute_plugin_function_v2 ... > /dev/null 2>&1 || status_exit_code=$?
  ```

  The `||` short-circuit prevents `set -e` from triggering on non-zero exit while
  still capturing the actual exit code.

### Fixed - systemd state mismatch after manual stop/start

- **`start_connector()` and `stop_connector()` (`src/bin/oradba_dsctl.sh`)**:
  add systemd delegation when a service unit exists and `INVOCATION_ID` is not
  set (i.e. not already running inside a systemd unit).

  **Root cause:** The service uses `Type=oneshot + RemainAfterExit=yes`. When
  a connector is stopped manually via `oradba_dsctl.sh stop`, systemd retains
  `active (exited)` state because it only tracked the `ExecStart` script exit,
  not the live process. `systemctl is-active` therefore returned `active` for a
  stopped connector.

  **Fix:** A new helper `get_systemd_service_name()` derives the unit name from
  the connector home path basename (`oracle_datasafe_<dir>.service`). If
  `systemctl cat <unit>` succeeds and `INVOCATION_ID` is unset, the function
  delegates to `sudo systemctl start/stop` instead of calling cmctl directly.
  When systemd calls `ExecStart`/`ExecStop`, it sets `INVOCATION_ID`, so
  `oradba_dsctl.sh` called from within a service unit invokes cmctl directly
  with no infinite-delegation loop.

  `status` always checks the actual port state - it never delegates to systemd.

## What Was Added

### `services` subcommand

- **`show_services()` and `services` action (`src/bin/oradba_dsctl.sh`)**: new
  subcommand that lists all installed `oracle_datasafe_*.service` units with
  ENABLED, STATE, and registry ALIAS columns.

  ```bash
  oradba_dsctl.sh services
  # SERVICE                                    ENABLED    STATE                ALIAS
  # oracle_datasafe_exacc-wob-vwg-ha1.service  enabled    active (exited)      dscon1
  # oracle_datasafe_exacc-wob-vwg-ha3.service  enabled    inactive (dead)      dscon3
  ```

- **`dssvc` alias (`src/etc/oradba_standard.conf`)**: shell shortcut for
  `oradba_dsctl.sh services`.

2. Run Docker integration test and record results in `doc/v1.0.0-readiness.md`

5. Create `doc/releases/v1.0.0.md`
6. Check `doc/review/risk-register.md` and `technical-debt-register.md` - open
   items become GitHub issues, then delete `doc/review/`.
7. Tag: `git tag v1.0.0 HEAD`
8. Push tag and let the release workflow create the GitHub Release
