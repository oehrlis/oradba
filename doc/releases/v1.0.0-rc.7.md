# Release Notes v1.0.0-rc.7

<!-- markdownlint-disable MD013 MD060 -->

**Release Date:** 2026-07-03
**Type:** Release Candidate (Bugfix RC - M10)
**Soak window:** 2026-07-03 to 2026-07-10 (same target as rc.1 through rc.6)

## Overview

Version 1.0.0-rc.7 fixes two additional `${!var}` indirect-expansion bugs under
`set -u`, the same root cause as the `extensions.sh` fix in rc.6. No API changes.

---

## What Changed Since rc.6

### Fixed - unsafe indirect expansion in `oradba_setup.sh`

- **`src/bin/oradba_setup.sh`** `show_extensions()`: changed `${!config_var}` to
  `${!config_var:-}` in the extension enabled/disabled check.

  **Root cause:** Under `set -u`, indirect expansion `${!config_var}` throws
  "unbound variable" when the env var `ORADBA_EXT_<NAME>_ENABLED` is not exported.
  The `:-` default returns an empty string; the `== "false"` check still works
  correctly (empty != "false" → extension stays enabled).

### Fixed - unsafe indirect expansion in `oradba_common.sh`

- **`src/lib/oradba_common.sh`** `verify_oracle_env()`: changed `${!var}` to
  `${!var:-}` in the missing-variable detection loop.

  **Root cause:** The loop iterates over `("ORACLE_SID" "ORACLE_HOME")` and checks
  `[[ -z "${!var}" ]]` to detect unset variables. Under `set -u`, the expansion itself
  throws "unbound variable" before the `-z` check runs — crashing instead of adding
  the variable to `missing_vars`. The `:-` default preserves the detection semantics.
