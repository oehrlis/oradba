# Release Notes v1.0.0-rc.6

<!-- markdownlint-disable MD013 MD060 -->

**Release Date:** 2026-07-01
**Type:** Release Candidate (Bugfix RC - M10)
**Soak window:** 2026-07-01 to 2026-07-10 (same target as rc.1 through rc.5)

## Overview

Version 1.0.0-rc.6 fixes three bugs in the installer infrastructure introduced or
uncovered during rc.5 testing. No functional changes to the Data Safe connector
management; no API changes.

---

## What Changed Since rc.5

### Fixed - SHA256 checksum verification failure in `oradba_install.sh --update --github`

- **`scripts/build_installer.sh`**: SHA256 checksum files now contain bare filenames
  (`oradba-1.0.0-rc.6.tar.gz`) instead of path-prefixed names (`dist/oradba-…`).

  **Root cause:** `sha256sum dist/oradba-1.0.0-rc.5.tar.gz` records the path as-is.
  The installer downloads the tarball to a flat temp directory and runs
  `shasum -c oradba-…tar.gz.sha256` from there. It looks for
  `dist/oradba-1.0.0-rc.5.tar.gz` relative to `$TEMP_DIR`, which does not contain
  a `dist/` subdirectory — verification always failed.

  **Fix:** Generation is now `(cd "$DIST_DIR" && sha256sum "${DIST_TARBALL##*/}")`,
  producing `<hash>  oradba-1.0.0-rc.6.tar.gz` with no path prefix.

### Fixed - RC version comparison in `version_compare()`

- **`src/bin/oradba_install.sh`**: `version_compare()` rewriiten to handle
  pre-release suffixes correctly.

  **Root cause:** `.`-based splitting of `1.0.0-rc.5` produced the array
  `["1","0","0-rc","5"]`. The suffix-strip `%%-*` reduced element [2] to `"0"` for
  every RC, so `1.0.0-rc.5` and `1.0.0-rc.2` both compared as `1.0.0` — equal.
  Likewise `1.0.0` and `1.0.0-rc.5` compared as equal.

  **Fix:** The function now separates the release string (`1.0.0`) from the
  pre-release suffix (`rc.5`) before comparing. Stable beats any pre-release;
  same-type RC numbers are compared numerically.

### Fixed - misleading "New version: 1.0.0-rc.2" in `--update --github`

- **`src/bin/oradba_install.sh`** `perform_update()`: stopped showing the running
  installer's own version as "New version" before the tarball is downloaded.

  **Root cause:** `perform_update()` is called before `TEMP_DIR` is created and
  before the tarball is downloaded. With no `VERSION` file available, `new_version`
  defaulted to `INSTALLER_VERSION` (the old installed value), producing
  "New version: 1.0.0-rc.2" and "Already running latest version" even when
  rc.5 was available on GitHub. The "already latest" check ran in a subshell
  (`$(perform_update …)`), so `exit 0` only exited the subshell — the installation
  continued anyway, but without the backup that `perform_update` was also
  responsible for creating.

  **Fix:** The version comparison block is skipped when no `VERSION` file is found.
  Embedded mode is exempt: `INSTALLER_VERSION` is always the correct payload version,
  so `version_known=true` is forced for `INSTALL_MODE=embedded`. The actual upgrade
  banner is printed after extraction when `INSTALLER_VERSION` has been updated from
  the downloaded tarball.

### Fixed - BATS tests 46/47 regression (embedded update mode)

- **`src/bin/oradba_install.sh`** `perform_update()`: tests "update skips when
  same version without force" and "update proceeds with --force flag" use the
  embedded installer (no `--github`/`--local`). The rc.6 `version_known` logic
  incorrectly treated embedded mode the same as github mode (skip comparison).
  Fixed by adding `INSTALL_MODE == "embedded"` check to force `version_known=true`.

### Fixed - `oradba_dsctl.sh` fallthrough when sudo unavailable

- **`src/bin/oradba_dsctl.sh`** `start_connector()` and `stop_connector()`:
  when a service unit exists but `sudo systemctl start/stop` fails (sudo not
  configured or user not in sudoers), log a `WARN` and continue to direct `cmctl`
  invocation instead of returning error code 1.

  **Behaviour matrix:**

  | Condition | Behaviour |
  |-----------|-----------|
  | No service defined | cmctl direct, no warning |
  | Service defined, sudo works | systemctl, systemd state stays in sync |
  | Service defined, sudo fails | WARN + cmctl direct, systemd state may drift |
  | Running inside systemd (`INVOCATION_ID` set) | cmctl direct, no delegation |
  | `systemctl` binary not found | cmctl direct, no warning |

## How to Promote to v1.0.0

After the soak window (not before 2026-07-10):

1. Confirm no Critical/High regressions filed against v1.0.0-rc.6
2. Run Docker integration test and record results in `doc/v1.0.0-readiness.md`
3. Mark remaining open items `[x]` in `doc/v1.0.0-readiness.md`
4. Set `VERSION` to `1.0.0` and add `## [1.0.0]` date to `CHANGELOG.md`
5. Create `doc/releases/v1.0.0.md`
6. Check `doc/review/risk-register.md` and `technical-debt-register.md` - open
   items become GitHub issues, then delete `doc/review/`.
7. Tag: `git tag v1.0.0 HEAD`
8. Push tag and let the release workflow create the GitHub Release
